### TPC-H Q1/Q3/Q6 — same-box head-to-head vs Apache Druid 35.0.1

**Current result (2026-07-11; AMD Ryzen 9 9950X x86_64; SF10 = 60 M
rows; Druid 35.0.1 single-server `large` profile, 31 processing
threads, all result/segment caching disabled; identical generated
data with rollup disabled; results byte-identical; median of ≥7
FerroDruid / 12 Druid iterations):**

| Query | FerroDruid | Druid 35 | Speedup |
|---|---:|---:|---:|
| Q1 — groupBy `(l_returnflag, l_linestatus)`, full range | **28.3 ms** | 338 ms | **12.0×** |
| Q3 — topN `l_orderkey` by revenue, 1-month interval | **1.01 ms** | 17 ms | **16.8×** |
| Q6 — filtered timeseries revenue, 1-year interval | **2.56 ms** | 14 ms | **5.5×** |

Conditions that must travel with these ratios: same box, single-node
Druid (not a distributed cluster); FerroDruid runs a single in-memory
segment in-process while Druid processes 21 segments over HTTP. The
measured Druid HTTP+serialize floor is ~3.4 ms — subtracting it, the
wins remain ~12× / ~13× / ~4×. Treat this as an honest
engine-vs-engine same-data figure, not a same-segmentation one; a
multi-segment FerroDruid re-run and a cloud-hardware corroboration
are tracked follow-ups. Correctness: Q1/Q3/Q6 results verified
byte-identical between the two engines on 60 M rows, plus
`crates/ferrodruid-query/tests/tpch_query_tests.rs` golden matches.
Parallel f64 summation is order-non-deterministic (an explicit
product decision matching Druid's parallel aggregation); count /
integer aggregates and row-set membership are exact.

History: the first head-to-head (2026-07-01, pre-optimization, both
engines sequential) measured FerroDruid 32×–2994× *slower* at SF10;
the 2026-07-11 result reflects the optimization work that followed
(in-scan parallelism over L1-tiled row chunks, dense accumulator
arrays, compiled typed filters, interval elision on sorted time).
Full evidence for both runs is retained in the vendor evidence pack
(available on request).

### Wave 47-D divergences (real findings)

These were surfaced by the diff harness; each was confirmed reproducible
across all 7 Druid versions tested.  Findings 2-5 and 7 closed in
the same wave; Findings 1 and 6 are deferred to a follow-up wave
(see "Deferred — next wave" below).

1. **Window functions silently elided in SQL planner** (8 queries).
   FerroDruid's parser accepted `OVER (...)` syntax but the planner
   did not emit a window-aware native plan — the result rows
   omitted the window-function column and used unspecified row order.
   Druid 31-36 returns the windowed result; Druid 30 returns `[]`
   for window queries (upstream `enableWindowing` not enabled by
   default in 30.x).  **Status: closed (Wave 47-D §1)** — added a
   native `Window` query type (`crates/ferrodruid-query/src/window.rs`)
   that wraps a `Scan`, evaluates `ROW_NUMBER` / `RANK` / `DENSE_RANK`
   / `LAG` / `LEAD` / `SUM(col) OVER ()` / `AVG(col) OVER ()` per
   partition with the window's own `ORDER BY`, then re-sorts the
   visible output by the outer SQL `ORDER BY`.  Planner now lowers
   any `func(...) OVER (...)` projection to this query type.
   Frame-clause variants (`ROWS BETWEEN ...`) and multi-column
   ordering on the same partition still default-frame; the 8
   harness queries do not exercise those.
2. **Native query: `dataSource: "name"` string shorthand rejected.**
   Druid accepts both the string and `{"type":"table","name":...}`
   forms; FerroDruid only accepted the verbose form.
   **Status: closed** — `DataSource` deserialiser accepts a bare
   string as a `Table { name }` (Wave 47-D §2 fix).  Serialisation
   continues to emit the canonical tagged form for stability.
3. **Native query: `dimension: "col"` string shorthand rejected.**
   FerroDruid required the full `DimensionSpec` object.
   **Status: closed** — `DimensionSpec` deserialiser now accepts a
   bare string `"col"` and an object with `outputName`/`outputType`
   omitted (defaulting to the dimension name and `STRING`).
4. **Native query: `metric: "name"` string shorthand rejected.**
   FerroDruid required `{"type":"numeric","metric":"name"}`.
   **Status: closed** — `TopNMetricSpec` deserialiser accepts a bare
   string as a `Numeric { metric }`.
5. **TopN granularity=all: bucket timestamp uses 1970 epoch.**
   Druid emits `timestamp: "<interval-start>"`; FerroDruid emitted
   `timestamp: "1970-01-01T00:00:00.000Z"`.
   **Status: closed** — TopN executor stamps the single bucket with
   the earliest interval start when granularity is `all`.  All 4
   pre-existing TopN queries now report `deep=true shape=true`.
6. **TIMESERIES sparse-bucket emission divergence.**
   For granularities finer than a day, Druid emits an entry for
   every gran-aligned bucket between the first and last non-empty
   bucket within each segment-granularity macro-block (with
   `result: {"col": null}` for empty buckets); FerroDruid skipped
   empty buckets entirely.  At granularity `hour` over a 3-day
   interval, Druid returns 35 rows (13 + 13 + 9 per day), FerroDruid
   returned 8 non-empty rows only.
   **Status: closed (Wave 9.4)** — the executor now expands the
   non-empty bucket set with synthetic empty buckets before
   serialising.  Because FerroDruid currently produces a single
   segment per ingestion task, the per-`segmentGranularity` fill
   that Druid does is approximated by grouping buckets under the
   next-coarser natural granularity (hour→day, minute→hour, ...) and
   filling each group from its own min to max.  This reproduces the
   harness's 35-row hour-over-3-day shape exactly.  Empty-bucket
   defaults match Druid's wire format: `0` for `count`, `null` for
   `*Sum` / `*Min` / `*Max` / `*First` / `*Last`.
   `skipEmptyBuckets:true` short-circuits the fill and preserves the
   compact "non-empty only" emission for clients that opt out.
7. **`longSum` aggregator on metric column returns 0.**
   `{"type":"longSum","fieldName":"added"}` used to return
   `total_added=0` for every bucket because `LongSumAggregator`
   only matched `Value::as_i64()` and the metric column was
   ingested as `Double` / `f64` (`100.0`, `200.0`, ...).
   **Status: closed** — the aggregator now coerces via `as_f64`
   and truncates toward zero, mirroring Druid's
   `Numbers.parseLong` semantics.  `ts_longsum_day` harness query
   returns 575/600/200 deep-matching Druid.
8. **Native query top-level field-order differs.**
   Druid returns `{"timestamp":..., "result":...}`; FerroDruid
   returns `{"result":..., "timestamp":...}`.  JSON-equality clients
   are unaffected (objects are unordered in JSON), but text-diff
   clients see this.  **Status: cosmetic** — left as documented
   evidence.

legacy-compat assessment):
